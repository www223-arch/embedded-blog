## 项目概述

这是一个基于 Vite + TypeScript 的个人博客项目，采用现代化设计风格，包含：
- 首页（Hero 区域 + 导航舞台 + 粒子特效）
- 项目作品展示
- 技术文档库
- 个人分享模块

### 技术栈

- **框架**: Vite + TypeScript
- **样式**: CSS3 + CSS 动画
- **内容管理**: Python + YAML 自动化生成
- **数据验证**: Zod
- **特效**: Canvas 粒子系统

## 模块设计

### 1. 首页模块 (`src/features/home/`)

#### 文件结构
```
src/features/home/
├── view.ts           # 首页视图
├── fx.ts            # 粒子系统
└── typewriter.ts    # 打字机效果
```

#### 粒子系统 (`fx.ts`)
- **类**: `HomeParticle`
- **主要功能**:
  - 粒子初始化
  - 粒子连线
  - 鼠标交互（驱散效果）
  - 智能速度调节
  - 防僵住机制
- **关键参数**:
  - `particleCount`: 粒子数量
  - `connectDist`: 连线距离
  - `repelRadius`: 驱散半径

#### 打字机效果 (`typewriter.ts`)
- **主要功能**:
  - 逐个字符打印
  - 循环切换身份标签
  - 应用不同样式类
- **身份样式映射**:
  - Engineer → `.role-engineer` (蓝青色渐变)
  - Photographer → `.role-photographer` (暖橙色)
  - Dreamer → `.role-dreamer` (粉紫色渐变)

### 2. 内容数据模块 (`src/content/`)

## 如何拓展功能

### 1. 添加新的内容类型

#### 步骤 1: 定义数据结构
在 `src/content/schema.ts` 中添加新的 Zod schema：

```typescript
export const yourNewSchema = z.object({
  id: z.string(),
  title: z.string(),
  // 添加其他字段
});

export type YourNewType = z.infer<typeof yourNewSchema>;
```

#### 步骤 2: 创建内容文件夹
在 `content/` 下创建新文件夹，比如 `content/your-new-type/`

#### 步骤 3: 修改 Python 脚本
在 `generate_content.py` 中添加：
- 加载函数
- TypeScript 生成函数
- 在 main() 中调用

#### 步骤 4: 创建页面视图
在 `src/features/` 下创建新模块

### 2. 添加新的页面路由

参考现有页面结构：
- 项目作品: `src/features/projects/view.ts`
- 技术文档: `src/features/docs/view.ts`
- 个人分享: `src/features/life/view.ts`

### 3. 自定义样式主题

在 `src/style.css` 的 `:root` 中添加 CSS 变量：

```css
:root {
  --your-color: #abc123;
  --your-shadow: 0 10px 30px rgba(...);
}
```

### 4. 添加新的动画效果

参考现有动画实现：
- Reveal 动画 (滚动出现)
- Hover 动画 (悬浮效果)
- 粒子动画 (Canvas)

### 5. 拓展粒子系统

在 `src/features/home/fx.ts` 中修改：
- 添加新的粒子行为
- 调整交互参数
- 自定义视觉效果

#### Python
- 遵循 PEP 8
- 添加错误处理
- 使用 UTF-8 编码

### 重要：避免乱码的安全策略

#### ?? 警告：Windows 中文编码问题

在 Windows 系统上编辑包含中文字符的文件时，极易出现编码乱码问题。请严格遵循以下安全策略：

#### 策略 1：Git Restore 为先

**在编辑任何文件前，先运行 git restore 确保文件初始状态正确：**

```bash
# 恢复单个文件
git restore src/style.css
git restore src/features/projects/view.ts
git restore src/app/bootstrap.ts

# 或者恢复多个文件
git restore src/style.css src/features/projects/view.ts
```

#### 策略 2：读取确认再编辑

**编辑任何包含中文的文件前，必须先读取确认：**

1. 用 Read 工具读取完整文件
2. 确认中文内容正常显示
3. 只编辑需要修改的部分
4. 完全保留中文内容原样

#### 策略 3：纯英文内容安全区

**只在文件末尾添加纯英文内容，避免触碰中文部分：**

- 样式文件：在 `/* Modal System Styles */` 之后添加新样式
- TypeScript 文件：只修改纯英文变量名、函数名
- 避免在包含中文的代码块内进行编辑

#### 策略 4：如果出现乱码怎么办？

**立即停止，用 Git 恢复：**

```bash
# 1. 立即恢复乱码的文件
git restore <乱码的文件名>

# 2. 重新开始，严格按照安全策略
```

#### 策略 5：分文件编辑

**尽量不要同时编辑多个包含中文的文件：**

- 一次只编辑一个文件
- 编辑完确认正常后再编辑下一个
- 频繁用 Read 工具检查状态

## 重要：架构问题反思与改进

### ? 问题记录（2026-04-01）

#### 现象
- 新增功能时频繁引入 BUG
- 修改一个功能影响其他功能
- 样式混乱、卡片透明、图片显示异常等

#### 根本原因

**1. 样式文件耦合严重**
- ? 所有样式混在一个 `style.css` 文件中
- ? 基础样式、组件样式、页面样式没有分层
- ? 修改风险极高，牵一发而动全身

**2. 缺乏命名空间**
- ? 类名容易冲突
- ? 没有明确的样式作用域

**3. 修改方式不安全**
- ? 随意修改已有样式
- ? 没有遵循开闭原则（OCP）

#### 架构改进建议（待实施）

**1. 样式模块化**
```
src/styles/
├── base.css        # 基础样式（重置、变量）
├── components.css  # 可复用组件样式
├── pages/          # 页面特定样式
│   ├── home.css
│   ├── projects.css
│   └── docs.css
└── utilities.css   # 工具类
```

**2. 采用 BEM 命名规范**
- Block: `.card`
- Element: `.card__image`
- Modifier: `.card--hover`

**3. 遵循开闭原则**
- ? 对扩展开放：在文件末尾添加新样式
- ? 对修改关闭：不随意修改已有样式
- ? 使用新增类而不是修改现有类

**4. 安全修改策略（强制执行）**
1. 编辑前先 `git restore`
2. 只在文件末尾添加新内容
3. 新增类名而不是修改现有类
4. 频繁 Read 检查状态
